# Step 3: 파이프라인 묶기 + 로그 (20분)

> Step 1에서 만든 `import_sales.py`와 Step 2에서 만든 `fetch_rates.py`를
> **하나의 명령**으로 돌리는 `run_pipeline.py`를 만듭니다.
> 그리고 실행 기록을 `logs/` 폴더에 남깁니다.

---

## 1단계: 왜 묶어야 하나요?

지금 상태:
```
Step A: py fetch_rates.py     ← 환율 갱신
Step B: py import_sales.py    ← 엑셀 반영
```

두 개를 매일 순서대로 돌려야 합니다. 하나 빼먹으면 데이터가 어긋나죠.

묶은 후:
```
Step C: py run_pipeline.py    ← A, B를 순서대로 자동 실행
```

**명령 하나**로 끝. 그리고 이 하나만 스케줄러에 등록하면 Step 4가 쉬워집니다.

---

## 2단계: AI에게 시키기

### Claude에게 입력

```
run_pipeline.py를 만들어줘.

요구사항:
1. fetch_rates.py와 import_sales.py를 함수로 import해서 순서대로 호출
   (subprocess 말고 Python import로 - CLAUDE.md에 이유 적혀있음)
2. 실행 순서: 환율 갱신 → 매출 import
3. 각 단계의 시작/끝/소요시간/결과를 logs/YYYYMMDD.log에 append
4. 한 단계라도 실패하면 exit code 1, 다 성공하면 0
5. 로그는 터미널에도 동시에 출력
```

### 실행

```
run_pipeline.py를 돌리고, 생성된 로그 파일을 보여줘.
```

## 3단계: 로그 왜 남기나요?

자동화의 무서운 점: **내가 안 볼 때 돌아갑니다.**

- 새벽에 실패했는데 아무도 모름
- 일주일 후에야 "어? 대시보드가 이상한데?"라고 발견
- 왜 틀어졌는지 찾기가 극도로 어려움

**로그가 없으면**: "뭐가 문제야?" → "모르겠어요"
**로그가 있으면**: "이 날짜 로그 보여줘" → "2026-04-05 API 타임아웃" → 원인 확인 3초

> **교훈**: 자동화는 "돌아가게 만드는 것" 절반, **"문제가 생겼을 때 알아챌 수 있게 만드는 것" 절반**입니다.

## 4단계: 여러 번 돌려보기

```
run_pipeline.py를 3번 연속으로 돌려줘. logs/YYYYMMDD.log가 덮어쓰여졌는지,
아니면 append 됐는지 확인해줘.
```

기대: 같은 파일에 **3번의 실행 기록이 시간순으로 누적**. 덮어쓰기 아님.

이유: 하루에 여러 번 돌릴 수 있어야 함. 오전 11:30 스케줄이 실패했을 때 강사가 수동으로 다시 돌리면, 그 기록이 **원본 로그를 덮지 않고 뒤에 추가**돼야 조사할 때 편합니다.

## 5단계: 일부러 실패시켜보기 (선택)

```
.env 파일의 KOREAEXIM_AUTHKEY를 일부러 잘못된 값으로 바꾸고
run_pipeline.py를 돌려줘. 그 다음 로그와 exit code를 보여줘.
끝나면 .env를 원래대로 복구해줘.
```

기대:
- `[환율 갱신] 실패: 인증 오류` 로그
- **`import_sales.py`는 그래도 진행** — 환율 테이블과 매출 테이블은 서로 의존하지 않으므로, 한 단계 실패가 다른 단계까지 막을 이유가 없음
- 매출 import은 정상 완료 로그
- 그래도 한 단계라도 실패했으니 **exit code 1**
- 로그에 "어디서 왜 터졌는지" 다 남아있어서 운영자가 모니터링으로 즉시 확인 가능

> **왜 fail-fast(앞 단계 실패하면 멈춤)가 아닌가요?** Step A의 출력이 Step B의 입력일 때만 fail-fast가 자연스럽습니다. 우리 파이프라인은 두 단계가 독립적이라, 환율이 실패해도 매출은 멀쩡히 들어가는 게 정직한 동작입니다. 부분 실패는 **로그 + exit code 1**로 알리고, 운영자가 모니터링으로 잡습니다.

> **교훈**: 실패를 "조용히" 처리하면 최악입니다. **시끄럽게 실패**하고, **왜 실패했는지 기록**하세요. exit code는 자동화 시스템이 "성공/실패"를 판정하는 유일한 신호입니다.

## 6단계: 어제 실패했으면 오늘 어떻게 하나요?

> Step 3의 핵심 질문입니다. 자동화가 실패했을 때 **복구가 간단한지**가
> 좋은 파이프라인과 나쁜 파이프라인의 차이입니다.

### 시나리오 A: 환율 API 장애 → 복구

**상황 설정** — Claude에게:

```
지금이 어제라고 가정하자. .env의 KOREAEXIM_AUTHKEY를 'BROKEN_KEY'로 바꾸고
run_pipeline.py를 돌려줘.
```

관찰:
- `[환율 갱신] 실패` 로그
- `[매출 import] 정상 완료` (독립 단계라 영향 없음)
- exit code 1
- `data/failure_YYYYMMDD.flag` 파일이 생성됨

**복구** — Claude에게:

```
.env를 원래 키로 복구하고, run_pipeline.py를 다시 돌려줘.
그리고 exchange_rates 테이블에 오늘 날짜 환율이 들어갔는지 SELECT 해줘.
monthly_sales 행 수가 그대로인지도 확인해줘.
```

관찰:
- 환율 정상 갱신
- 매출 행 수 동일 (멱등성!)
- exit code 0
- failure flag 자동 삭제
- (week05 대시보드를 띄워뒀다면) 빨간 배너가 사라짐

### 시나리오 B: 엑셀 파일 누락 → 복구

```
법인_US01_미국.xlsx를 임시로 이름을 바꿔서 파이프라인이
미국법인을 못 읽게 해줘. 그 상태로 run_pipeline.py 돌려줘.
```

관찰:
- 7개 파일만 처리, US01 데이터는 DB에 기존 값 유지 (삭제되지 않음)
- "UPSERT 42건" (미국 6건 빠짐)

```
파일명을 원래대로 돌리고 run_pipeline.py 다시 돌려줘.
```

관찰:
- 8개 파일 모두 처리, UPSERT 48건
- monthly_sales 총 행 수 그대로 (48)

### 왜 "그냥 다시 돌리면 된다"가 가능한가요?

**멱등성** 덕분입니다. Step 1에서 배운 UPSERT가 여기서 빛을 발합니다:
- 이미 있는 데이터 → 덮어쓰기 (중복 아님)
- 없는 데이터 → 새로 넣기

만약 멱등성이 없었다면?
- "다시 돌렸더니 매출이 두 배로 뻥튀기"
- "어제 넣은 거 먼저 지워야 하는데 어떤 거가 어제 건지 모르겠어요"
- → 복구가 아니라 **추가 사고**

> **교훈**: 멱등성은 "편의 기능"이 아니라 **복구 가능성의 근간**입니다. "실수로 한 번 더 돌려도 괜찮다"는 말이 가능해지는 순간, 자동화는 무서운 것에서 **편한 것**으로 바뀝니다.
