## 3단계: 현실 세계의 함정들

이 API를 그냥 쓰면 **4가지 함정**이 있습니다. CLAUDE.md에 미리 적어둬서 Claude가 처리하지만, **여러분은 알고 있어야 합니다** — 실무에서 다른 API를 쓸 때도 똑같이 만나니까요.

### 함정 1: JPY는 100엔 단위로 옵니다

API가 주는 값:
```json
{ "cur_unit": "JPY(100)", "deal_bas_r": "920.00" }
```

이게 "1엔에 920원"이 아니라 **"100엔에 920원"**입니다. 우리 DB는 1엔 기준이므로 `÷ 100`을 해야 `JPY = 9.2`가 됩니다.

> **교훈**: API 문서를 꼼꼼히 읽지 않으면 **환율이 100배 튀는 대형 사고**가 날 수 있습니다.

### 함정 2: CNY가 CNH로 옵니다

API는 `CNY`가 없고 `CNH`(중국 역외위안)만 있습니다. 실무에선 거의 동일한 환율이라 **우리는 `CNH`를 받아서 `CNY`로 저장**합니다.

> **교훈**: 같은 통화라도 시장·거래소에 따라 다른 코드를 씁니다. 매핑 규칙을 하네스에 적어둬야 합니다.

### 함정 3: VND는 아예 없습니다

한국수출입은행은 베트남 동(VND)을 제공하지 않습니다. 그렇다고 프로그램이 죽으면 안 됩니다.

**폴백 규칙**:
1. DB에 있는 가장 최근 VND 환율을 재사용
2. 그것도 없으면 하드코딩 상수 `0.056` 사용
3. 로그에 "VND: 폴백값 사용" 경고

> **교훈**: "없으면 뭘 할지"를 미리 정해두지 않으면 에러만 나고 끝납니다. 폴백은 자동화의 필수.

### 함정 4: 주말·공휴일은 빈 응답

토요일에 호출하면:
```json
[]
```

에러가 아니라 **"데이터 없음"**입니다. 그렇다고 "오늘 환율이 0원"이 되면 큰일나죠.

**폴백 규칙**: 직전 영업일의 환율을 오늘 날짜로 복사합니다. 로그에 "영업일 아님, 재사용" 기록.

> **교훈**: 외부 서비스는 "주는 대로 받아서 쓰면 되지"가 절대 안 됩니다. **예외 상황을 미리 정의**해두는 게 80%입니다.

## 5단계: 멱등성 다시 확인

환율도 멱등성이 있어야 합니다. 같은 날 두 번 돌려도 DB에 2개가 아니라 1개여야 합니다.

```
fetch_rates.py를 한 번 더 돌려줘.
exchange_rates 테이블의 오늘 날짜 행 수가 8개인지 확인해줘.
```

**이게 자연스럽게 보장되는 이유**: `exchange_rates`의 PK가 `(currency, rate_date)` 복합키라서, 같은 `(USD, 2026-04-09)`를 두 번 INSERT하면 두 번째는 **덮어쓰기**가 됩니다.

week05에서 DB를 설계할 때 이 복합키를 박아둔 게, 여기서 **무료로 멱등성을 받게** 되는 겁니다.

> **교훈**: 좋은 DB 설계는 미래의 자기 자신에게 주는 선물입니다.

## 에러가 나면?

| 에러 | 원인 | 해결 |
|------|------|------|
| `KOREAEXIM_AUTHKEY 환경변수가 없습니다` | `.env` 미설정 | 1단계 다시 |
| `Connection timed out` | 네트워크 / 방화벽 | "잠시 후 재시도해줘" |
| `result != 1` | API 키 잘못됨 or 주말 | "에러 메시지 보여주고 어떻게 처리할지 알려줘" |
| 모든 환율이 이상하게 큼 | JPY ÷100 누락 | "CLAUDE.md 통화 매핑 규칙을 다시 읽고 고쳐줘" |
