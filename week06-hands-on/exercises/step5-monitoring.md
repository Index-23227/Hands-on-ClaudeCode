# Step 5: 실패하면 어떻게 알지? (15분)

> 자동화의 무서운 점: **내가 안 볼 때 돌아갑니다.**
> 새벽에 실패했는데 아무도 모르면? 일주일 후에야 "대시보드가 이상한데?"라고 발견합니다.
> 이번 단계에서는 **실패를 눈에 보이게** 만듭니다.

---

## 1단계: 지금 우리 파이프라인의 약점

Step 3까지 만든 파이프라인은:
- 실패하면 `logs/` 폴더에 기록하고 exit code 1을 반환
- 그런데 **누가 로그를 매일 열어보나요?**
- exit code 1을 **누가 확인하나요?**

> **교훈**: 로그는 "이미 문제를 알고 있을 때" 원인을 찾는 도구입니다. **문제를 발견하는** 도구가 아닙니다. 모니터링이 없으면 로그는 무용지물입니다.

---

## 2단계: 실패 알림 추가하기

### Claude에게 입력

```
run_pipeline.py에 다음 기능을 추가해줘:

1. 파이프라인이 실패하면:
   - data/failure_YYYYMMDD.flag 파일을 생성
   - 바탕화면에 '파이프라인_실패_YYYYMMDD.txt' 요약 파일 생성
   (다음날 아침 PC 켤 때 눈에 들어오게)

2. 파이프라인이 성공하면:
   - 오늘자 failure flag가 있으면 삭제 (재실행으로 복구됨)

flag 파일 경로는 week06-hands-on/data/ 아래에 만들어줘.
```

---

## 3단계: 시연 — 실패 → 알림 → 복구

### 3-1. 의도적 실패

```
.env의 KOREAEXIM_AUTHKEY를 'BROKEN'으로 바꾸고
run_pipeline.py를 돌려줘.
```

관찰:
- `[환율 갱신] 실패` 로그
- `data/failure_YYYYMMDD.flag` 생성 확인
- 바탕화면에 `파이프라인_실패_YYYYMMDD.txt` 파일 확인

### 3-2. 대시보드에서 확인

week05 대시보드를 띄워보세요 (`py ../week05-hands-on/app.py`).

기대: 대시보드 상단에 **빨간 배너**:
> **파이프라인 실패 감지** — 자동화 파이프라인이 마지막 실행에서 실패했습니다.

이게 핵심입니다. **매일 아침 대시보드를 여는 사람이라면, 빨간 배너를 보고 즉시 인지**합니다. 별도 모니터링 시스템 없이, 이미 보고 있는 도구에 알림이 뜨는 것.

### 3-3. 복구

```
.env를 원래 키로 복구하고 run_pipeline.py를 다시 돌려줘.
```

관찰:
- exit code 0
- failure flag 자동 삭제
- 대시보드 새로고침 → **빨간 배너가 사라짐**
- 바탕화면 txt는 남아있음 (수동 삭제 — "이런 일이 있었다"는 기록)

## 4단계: 왜 이렇게 만들었나요?

| 방법 | 장점 | 단점 |
|------|------|------|
| 슬랙/이메일 알림 | 즉각적 | 서버 설정 필요, 비전공자가 못 함 |
| Windows 이벤트 뷰어 | 내장 기능 | 비전공자가 안 봄 |
| 바탕화면 txt | PC 켜면 바로 보임 | 실시간 아님, 다음 날 아침에야 인지 |
| **대시보드 빨간 배너** | **이미 보고 있는 도구에 알림** | week05와 연동 필요 |

우리는 **바탕화면 + 대시보드**를 조합했습니다.
- 바탕화면: 다음 날 아침 PC 켤 때 인지 (개인)
- 대시보드: 팀원 누구든 열면 인지 (공유)

> **교훈**: 모니터링의 핵심은 **"이미 보고 있는 곳에 알리기"**입니다. 별도 도구를 새로 만들면 "그 도구를 누가 보지?" 문제가 다시 생깁니다.

---

## 여기까지 하면

- 파이프라인이 실패하면 **눈에 보이는 알림**이 뜹니다
- 재실행으로 복구하면 알림이 **자동으로 사라집니다**
- 모니터링 없는 자동화가 왜 위험한지 체감했습니다
